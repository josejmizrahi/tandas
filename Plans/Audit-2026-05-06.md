## 1. Lo que está sólido — no tocar

### 1.1 Codegen Swift↔TS está enforced en CI

`scripts/codegen/` (Deno) parsea enums marcados con `// @codegen:enum`,
regenera `ios/Tandas/Platform/Models/Generated/*+Codable.swift` y
`supabase/functions/_shared/types/*.ts`. Lefthook lo corre pre-commit
(`lefthook.yml`); GitHub Actions falla el PR si hay diff
(`.github/workflows/codegen.yml`).

Esto es lo más importante de toda la base. El riesgo P1 del Roadmap
("drift de tipos Swift↔TS, ya pasó 3 veces") está **resuelto**. No
tocar la infraestructura.

Decoders Swift usan `unknown(String)` para forward-compat; nunca
crashean por enum case desconocido.

### 1.2 Rule engine es puro y determinístico

`supabase/functions/_shared/ruleEngine.ts` (484 LOC):

- Lee solo del `RuleContext` que recibe
- Sin `Date.now`, sin RNG, sin estado mutable
- Tabla `phase_target` mapea cada tipo no implementado a la fase que
  lo va a implementar (`phase_2` para slot/rotation, `phase_3` para
  fund, `phase_4` para custom rule editor)
- Logs estructurados con `phase_target` cuando se encuentra evaluator
  no implementado — surfaces como signal real, no silent skip
- Tests unitarios en `ruleEngine.test.ts` (458 LOC)

Cumple los principios #5 (determinismo absoluto) y #8 (failure modes
predecibles) de la Vision sin hacks.

### 1.3 Event sourcing funciona

- `system_events` table con índice parcial sobre `processed_at IS NULL`
  para el cron `process-system-events`
- `record_system_event` RPC es la única vía de inserción
- `SystemEventEmitter` Swift swallows errors y loggea (nunca bloquea
  user-facing flow)
- 28 `SystemEventType` cases declarados; rule engine evalúa contra los
  V1, log estructurado para los V2+

Cumple principio #3 de la Vision.

### 1.4 Inbox + History son resource-agnostic

- `UserAction` table es polimórfica (`reference_id` UUID + `action_type`
  text). 9 `ActionType` cases declarados; UI renderiza por tipo
- `InboxCoordinator` consume cross-group sin asumir resource type
- `GroupHistoryCoordinator` filtra por `SystemEventType` arbitrario
- `HistoryItemPresentation` ya hace switch sobre todos los event types

### 1.5 Vote primitive genérica

- Tabla `votes` polimórfica desde 00020 (`vote_type` text + `reference_id`)
- 8 `VoteType` cases declarados (fineAppeal, ruleChange, ruleRepeal,
  memberRemoval, fundWithdrawal, roleAssignment, generalProposal,
  slotDispute)
- `start_vote` + `cast_vote` + `finalize_vote` todos polimórficos
- Migration 00023 cierra el edge case "1 persona decide en grupo de 2-3"
  con `quorum_min_absolute` + exclusion del infractor en fine_appeal

V1 solo emite `fine_appeal` pero la infra está lista para los otros 7.

### 1.6 Governance se evalúa via `GovernanceService`

`actor GovernanceService` con `canPerform(action, member, in: group, context:)`
es el único punto de decisión. Cumple principio #6. Cero
`if member.role == "admin"` regado en código.

### 1.7 Anonimato de votos enforced por RLS

- Vista `appeal_vote_counts` agregada (no expone ballots individuales)
- Policy `appeal_votes_read_own_or_aggregate` solo retorna ballots propios
- Misma estructura para `vote_casts` en sistema genérico

Cumple principio #7. No es convención cliente — es enforcement DB.

### 1.8 Sentry shipped

iOS + 13 edge functions con Sentry wrapper. PostHog + dSYM upload son
follow-ups P3.

## 2. Lo que está obsoleto — debe morir antes de Fase 2

### 2.1 `DinnerRecurringTemplate.TemplateID` enum

`ios/Tandas/Templates/DinnerRecurring/DinnerRecurringTemplate.swift:15-20`
declara enum dentro del archivo Swift de UN template específico que
lista IDs de TODOS los templates futuros:

```swift
public enum TemplateID: String, Sendable {
    case dinnerRecurring = "recurring_dinner"
    case sharedResource  = "shared_resource"  // Fase 2 placeholder
    case rotatingSavings = "rotating_savings" // Fase 3 placeholder
    case custom          = "custom"           // Fase 4 placeholder
}
```

Viola directamente principio #2 de la Vision ("templates como
configuración, no como código"). Cuando llegue Fase 2 alguien va a
editar el archivo del template viejo para agregar uno nuevo. Smell
crítico. Y duplica la `templates` table que ya tiene seed rows.

**Acción**: mover los IDs a constante neutral (`TemplateRegistry.knownIds`)
o eliminar el enum y leer del registry.

### 2.2 `appeal_votes` + `appeal_vote_counts` legacy

Migration 00023 (`appeal_voting_v2`) movió fine appeals al sistema
genérico `votes`/`vote_casts` con `vote_type='fine_appeal'`. Pero las
tablas `appeals`, `appeal_votes`, y la view `appeal_vote_counts` (todas
de 00014) **siguen existiendo y usándose**. Coexisten dos sistemas de
votación.

**Acción**: cleanup migration que reemplaza `appeals` por
referencia tipada a `votes.id`; deprecar `appeal_votes` y
`appeal_vote_counts`.

### 2.3 Columnas legacy en `rules` table

`Rule.swift:8-10` declara explícitamente:

> Persisted in `public.rules`. The legacy columns (`code`, `title`,
> `trigger`, `action`, `enabled`, `status`) are kept for backwards compat
> and dropped in a posterior sprint after migration paridad.

El paridad ya pasó. Drop migration pending. Mientras existan, el schema
es ambiguo y los repos tienen que ignorar columnas viejas.

### 2.4 `vote_ballots` (legacy)

Listado en `UICompleteCoverage.md:94`. No se usa en código nuevo. Drop.

### 2.5 `events_view` projection-only

Diseñada en 00014 como "Sprint 1a alias only" para volverse UNION con
`resources` filtered to `type='event'` y eventualmente leer solo de
`resources`. Hoy enmascara el dual-write nunca activado (ver §4.1).

## 4. Lo que necesita cambiar — riesgos estructurales

> Estos son los que **rompen Fase 2/3 si no se atienden ANTES**.

### 4.1 Riesgo 1 — `fines.event_id` y `fine_review_periods.event_id` event-locked

```sql
-- 00007_phase4_fines.sql + 00014_platform_foundation.sql
fines.event_id              → references public.events(id)
fine_review_periods.event_id → references public.events(id)
```

Cuando llegue Fase 3 (slots) o Fase 4 (funds), una multa por declinar
slot o no contribuir al fondo **no tiene `event_id`**. El rule engine ya
lo nota: `ruleEngine.ts:327` hace `event_id: target.resource_id` —
está pasando un resource_id pero la columna se llama event_id.

**Acción ANTES de Fase 2:**

1. Migration: `alter table fines add column resource_id uuid references resources(id) on delete cascade nullable`. Backfill desde `event_id`. Idem `fine_review_periods`.
2. Update `propose_fine` RPC + rule engine sink para escribir ambas columnas.
3. V2: drop `event_id`.

**Costo**: 2-3 días. **Si no se hace**: Fase 2 descubre el bug, refactor en caliente con datos en prod.

### 4.2 Riesgo 2 — Templates hardcoded en Swift

Decisión #4 de Vision dice "solo mover a Templates/DinnerRecurring/, no
renombrar". OK como táctica corta. Pero principio #2 dice "templates como
configuración, no como código".

Hoy `DinnerRecurringTemplate.defaultRules(groupId:)` es código Swift
que retorna `[Rule]`. Cuando llegue Fase 6 (editor visual + import/export
de templates), no se puede serializar/deserializar Swift code a JSON.

**Acción ANTES de Fase 2:**

- Mover los 5 default rules de `DinnerRecurringTemplate.swift` a
  `templates` table (`config.defaultRules` jsonb). Ya hay seed migration
  00015. Verificar que el seed tiene la verdad y eliminar duplicación
  Swift.
- Eliminar enum `TemplateID` de `DinnerRecurringTemplate.swift`.
- Convertir `defaultRules(groupId:)` en helper que materializa desde
  `Template.config.defaultRules`.

**Costo**: 1-2 días. **Si no se hace**: vas a tener `SharedResourceTemplate.swift`
con su propio enum `TemplateID`, el patrón se replica.

### 4.3 Riesgo 3 — `GovernanceRules` struct no extensible sin migration

Hoy: 6 `whoCan*` campos hardcoded + 4 voting fields. Cuando Fase 4
traiga funds, vas a necesitar `whoCanWithdrawFromFund`, `whoCanContribute`.
Cuando Fase 5 traiga proposals, `whoCanCreateProposal`.

Cada uno requiere: agregar campo Swift + agregar default deserializer +
migration backfill `groups.governance` jsonb + coordinar Swift release
con DB release.

Patrón no escala. Contradice principio "templates como configuración".

**Acción**: convertir `GovernanceRules` en `[String: PermissionLevel]`
(jsonb directo) con conveniencia API. Cada acción se define por
`GovernanceAction.rawValue` (ya es string). Struct typed es opcional
para autocompletado.

**Cuándo**: no urgente para Fase 2 (Rotation no introduce nuevos whoCans).
**Antes de Fase 4**.

### 4.4 Riesgo 4 — `system_events` sin particionado ni archival

~100k grupos × 100 events/mes = 10M rows/mes. Query del cron sigue
rápida (idx parcial unprocessed). Pero queries de history se vuelven
lentas, costo Supabase escala lineal.

**Acción ANTES de Fase 1 launch**: documentar plan
(`Plans/SystemEventsArchival.md`).
**Acción ANTES de Fase 4**: implementar particionado mensual + cold
storage events > 90 días.

### 4.5 Riesgo 5 — `MemberRole` enum Swift typed

Mismo problema que `GovernanceRules`. Cuando Fase 5 introduzca roles
configurables (cada grupo define sus propios roles), el enum bloquea.

**V2 solution**: roles son `String` (raw value), validados contra
`roles` table per-group, deserializados tolerantemente. Enum se mantiene
para V1's fixed roles solo como helper.

**Cuándo**: documentar en `Plans/RolesV2.md` ahora. Implementar antes
de Fase 5.

### 4.6 Riesgo 6 — `GroupModule.providedRules` usa display strings como IDs

`V1Modules.swift:23-29`:

```swift
providedRules: [
    "Llegada tardía",          // display string, NOT id
    "No confirmó a tiempo",
    ...
]
```

Si se localiza la app o cambia el copy de una regla, los modules
apuntan a IDs que no existen.

**Acción**: rule_id estable (slug o UUID) por separado del display name.
**Costo**: 4-6h. Hacer junto con Riesgo 2 (mover templates a data).

### 4.7 Riesgo 7 — Dos `ResourceProtocol` paralelos

`Platform/Resources/ResourceProtocol.swift` (UI dispatch) vs
`Platform/Models/Resource.swift` (data with Codable + status + timestamps).

Doc declara "distinct" intencionalmente. Pero cuando llegue
`SlotResource` (Fase 3), va a tener que conformar a ambos, y los
invariantes (resourceType consistente, id consistente) no están
enforced en un solo lugar. Bugs sutiles esperando a aparecer.

**Acción decidida 2026-05-06**: **convergencia explícita en Sub-fase D
o E de Fase 0.5**, no esperar a que aparezca el problema con Slot.
Acta clara de "estos son el mismo concepto" — el data protocol es la
verdad, el UI protocol o desaparece o pasa a ser un alias enforced.

correctas en código, codegen real y enforced, rule engine determinístico
con phase_target mapping, governance verificable, anonimato enforced

   replique.

doc archival plan. Eso cierra Fase 0 verdaderamente.

**Lo programable (no crítico ahora pero sí documentar)**:
GovernanceRules jsonb, MemberRole configurable, system_events archival
implementado. Documentar planes; ejecutar antes de las fases que los
necesitan.

### 9.2 Razones de los tres ajustes vs propuesta original del audit

1. **Templates a data sube a item #1**: 1-2 días y previene replicación
   del patrón en cualquier feature nueva de F0. OpenVotesView u otras
   features de F0 podrían heredar referencias indirectas al enum mal
   localizado.

2. **APNs real entra como P0 de F0 en paralelo**: sin notificaciones
   reales, "árbitro neutral que recuerda" no se cumple. Beta con grupo
   real arranca con UX rota; validación de producto se contamina. No
   es gate de otras features (paralelo), pero sí gate del beta.

3. **Convergencia `ResourceProtocol` explícita en Sub-fase D/E, no
   en Sub-fase E "probablemente"**: cuando llegue Slot en Fase 2, los
   dos protocolos van a estar replicados con invariantes no enforced.
   Bugs sutiles. Mejor converger ahora con acta clara.

### 9.3 Pausa pre-Fase 2 (compromiso)

Después de cerrar F0 + items pre-Fase 2: **NO arrancar Fase 2
inmediato**. 2-4 semanas usando V1 con grupo real.

Razón: la calidad técnica de ruul ya supera la de productos consumer
típicos. El riesgo del proyecto NO es técnico — es de validación de
producto. Lo que va a hacer triunfar o fallar a ruul no es si las
primitivas están bien diseñadas (lo están), sino si grupos reales lo
usan y vuelven.
