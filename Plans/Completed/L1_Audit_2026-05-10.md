# L1 Audit — Identity / Membership / Group / Rules

**Fecha:** 2026-05-10
**Trigger:** Founder feedback "siento que faltan cosas en las primeras layers"
**Audit method:** 3 paralelos Explore agents (rules, identity/profile, membership/governance)

---

## TL;DR

Tres hallazgos arquitectónicos. Uno de fondo (rules), dos de superficie (UI/UX gaps).

1. **[FONDO] Rules architecture es mono-nivel.** Hoy todas las reglas se siembran al crear el grupo via `seed_template_rules`. Eso bloquea Phase 2 (slot/booking/asset módulos no podrán inyectar sus reglas al activarse) y deja zombies al desactivar módulos.
2. **[SUPERFICIE] Identity layer shallow.** `EditProfileSheet` solo edita displayName. Avatar capturado en onboarding pero no editable. Phone capturado en `auth.users` pero NO sincronizado a `profiles.phone` (campo muerto). Sin notification preferences.
3. **[SUPERFICIE] Group profile fragmentado.** Settings disperso en 3 sheets (`GroupInfoSheet` + `GroupSettingsSheet` + `GovernanceSettingsView`). Editable: vocabulary, rotation, fines toggle, governance, members. **NO editable:** group name, description, cover image, frequency, invite code regen.
4. **[SUPERFICIE] Membership/Permissions consultivos, no enforced.** `has_permission` RPC existe (mig 00063), `GovernanceService.hasPermission()` lee localmente — pero los botones no se esconden por permission. Suspended/observer/guest states sin reachability. Sin role editor, sin remove member flow, sin archive group.

## Hallazgo 1 — Rules architecture (FONDO)

### Estado actual

**Schema:**
- `rules` (mig 00001 + 00014): solo `group_id` como scope. NO existe `module_key`, NO existe `resource_id`.
- `groups.governance` (mig 00019): jsonb con quorum/thresholds/permissions — esto SÍ es group-level y correcto.
- `groups.active_modules` (mig 00019): jsonb array de slugs activos.
- `modules.provided_rules` (mig 00060): array de slugs (referencias), NO definiciones.
- `templates.config.defaultRules` (mig 00021): array completo de definiciones, con campo `module` que existe pero es **documentación dead-code**.

**Flow actual:**
1. Founder crea grupo → `create_group_with_admin` (no toca rules).
2. iOS llama `seed_template_rules(template_id, group_id)` UNA VEZ.
3. RPC bulk-inserta TODAS las `defaultRules` con scope = group_id solamente.
4. `set_group_module(slug, enabled)` togglea `active_modules` array, **sin tocar rules**.

**Bugs derivados:**
- **B1.** Activar `slot_assignment` post-onboarding nunca siembra sus reglas. Phase 2 está bloqueada.
- **B2.** Desactivar `basic_fines` deja sus 5 reglas zombi en el grupo. Engine las sigue evaluando.
- **B3.** No hay forma de override por instancia (cena específica con regla distinta). Se necesitará para Trip/Asset.
- **B4.** No hay programmatic affinity rule→module. Solo se puede inferir por slug pattern.

### Tres niveles correctos

| Nivel | Scope | Storage | Lifecycle |
|---|---|---|---|
| **Group rules** | governance pura del grupo | `groups.governance` jsonb (✓ ya existe) | mutable via vote o founder |
| **Module rules** | comportamiento de capability | `rules` con `module_key` set | seed on `set_group_module(.., true)`, archive on `false` |
| **Resource rules** | override en instancia específica | `rules` con `resource_id` set | seed on resource create, delete on resource delete |

### Files que cambian (ver Phase A plan)
- New mig: add `module_key text`, `resource_id uuid`, `scope text` to `rules`
- New mig: move rule defs from `templates.config.defaultRules` → `modules.provided_rules_def jsonb`
- New mig: rewrite `seed_template_rules` (group-level only)
- New mig: add `seed_module_rules`, `archive_module_rules` RPCs
- New mig: extend `set_group_module` cascade
- New mig: backfill existing groups' rules with `module_key`
- iOS: `RuleRepository.swift`, `GroupsRepository.setModule`, `ModuleRegistry`

## Hallazgo 4 — Membership / Governance (SUPERFICIE)

### Roles & Permissions

- ✓ Schema: `groups.roles` jsonb (mig 00063) con default seed `founder` + `member`.
- ✓ Swift: `RoleDefinition.swift`, `Permission` enum (15 acciones, 9 V1 + 6 Phase 2-4).
- ✓ RPC: `has_permission(group_id, user_id, permission)` (mig 00063:135-186).
- ✗ UI: NO role editor; `assign_role` RPC pendiente Phase 5.
- ✗ Enforcement: `GovernanceService.hasPermission()` consultivo local; botones NO se esconden por permission.

### Membership states

| Estado | Reachable | Schema |
|---|---|---|
| founder | ✓ | role='founder' |
| admin (alias) | ✓ | role='founder' (legacy) |
| member | ✓ | role='member' |
| host | ✓ (per-event) | event-scoped, no group-level |
| suspended | ✗ implícito (active=false, sin UI) | `active` bool |
| invited | ✗ no tracked (PendingInvite es FE-only) | n/a |
| guest | ✗ no modelado | n/a |
| observer | ✗ no modelado | n/a |
| removed | ✗ implícito (active=false) | n/a |

### Group lifecycle

- ✓ Create (`create_group_with_admin`)
- ✓ Join (`join_group_by_code`)
- ✗ Archive
- ✗ Delete
- ✗ Leave (con o sin founder transfer)

## Decisión inmediata

1. Es el bug arquitectónico que el founder identificó.
