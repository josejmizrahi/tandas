> Fecha: 2026-05-07
> Estado: Input para propuesta de rediseño Fase 0
> Auditor: Claude Code (exhaustivo)

---

## 0. Resumen ejecutivo

**Hallazgos clave que requieren intervención inmediata:**

1. **Group switcher funciona pero es poco visible** — El menú contextual en HomeView es intuitivo (patrón iOS nativo) pero sin affordance visual en otras tabs. Al cambiar grupos, el cambio se propaga correctamente pero no hay feedback visual claro. Los usuarios no descubren fácilmente que pueden cambiar grupo desde el header.

2. **Loading states: ProgressView bare + sin jerarquía de carga** — Views como RulesView, OpenVotesListView, inboxTab muestran centrado `.ProgressView()` sin diferenciación entre initial load, refresh, y error. Existe `LoadingStateView` con skeletons pero NO se usa en vistas principales. El BootstrappingView (línea 277 de AuthGate.swift) es el único que usa ProgressView con tamaño adecuado.

3. **MainTabView es monolítica con 23 @State variables** — La navegación está en MainTabView con coordinators construidos lazily. Hay 4 NavigationStack separados (uno por tab) pero comparten AppState. El coordinador rebuild en onChange(activeGroupId) es correcto pero hace el cambio perceptible (tap delay + flicker).

4. **Design system es sólido pero poco usado en ciertas áreas** — Existen RuulColors, RuulTypography (bien jerarquizados), RuulSpacing (4/8/16/24 system) y RuulGlass. Pero: algunos placeholders usan valores hard-coded, no todas las views aplican Dynamic Type, y las transiciones de loading→content carecen de motion tokens (RuulMotion existe pero infrautilizado).

5. **Empty states + error states + retry logic fragmentados** — `EmptyStateView` existe y es bonito. Pero no todas las vistas lo usan (ej: algunos coordinators muestran solo isLoading=false sin fallback). Error handling es ad-hoc: algunos coordinators tienen `.error` property, otros no.

## 2. Group switcher (deep dive)

### ¿Cómo está hoy?

**State:**
- Vive en `AppState.activeGroupId` (línea 15, AuthGate.swift)
- Se persiste en UserDefaults (`ruul_active_group_id` key)
- Resuelve a `AppState.activeGroup` (fallback a groups.first si stale)

**UI:**
- **HomeView**: Menu contextual en el header (línea 112-162, HomeView.swift)
  - Patrón nativo iOS (como Calendar.app)
  - Muestra cada grupo + pending count badge
  - Checkmark en el active
  - Tap → `app.activeGroupId = group.id` → dismissMenu
  
- **MainTabView sheets**:
  - `GroupSwitcherSheet` (línea 89-97, MainTabView.swift)
  - Tap on "Cambiar de grupo" button en HomeView FAB
  - Lista grupos con initials avatar, nombre, vocabulario
  - "Crear grupo" / "Unirme con código" action rows
  - Visual: rounded rectangles, accent colors

**¿Qué "falla"?**

1. **Affordance débil en tabs diferentes** — Inbox, Rules, Profile no tienen acceso visual al group switcher. Solo una deep link si el usuario abre HomeView primero.

2. **No hay loader visual durante rebuild** — MainTabView.onChange(activeGroupId) → rebuildCoordinators() toma ~200ms (3 coordinators fetching). Durante ese tiempo el tab content puede parecer "congelado" sin transición.

3. **Falta visual feedback de "you are here"** — En HomeView Menu, el grupo activo tiene checkmark. Pero si el usuario está en Inbox tab viendo acciones de Grupo B, no sabe si su contexto cambió sin volver a HomeView.

4. **No hay "sticky group selector" visible** — Algunos apps (Slack, Figma) ponen el selector en una ubicación fixed. Ruul lo esconde en HomeView header → es un "hidden gem".

### Código actual

**AppState.**
```swift
// AuthGate.swift:15-32
var activeGroupId: UUID? {
    didSet {
        if let id = activeGroupId {
            UserDefaults.standard.set(id.uuidString, forKey: Self.activeGroupKey)
        }
    }
}
var activeGroup: Group? {
    if let id = activeGroupId, let g = groups.first(where: { $0.id == id }) {
        return g
    }
    return groups.first
}
```

**MainTabView coordinator rebuild:**
```swift
// MainTabView.swift:80-88
.onChange(of: app.activeGroupId) { _, _ in
    if let group = app.activeGroup else { return }
    await rebuildCoordinators(for: group)
}
```

**rebuildCoordinators:**
```swift

@MainActor
private func rebuildCoordinators(for group: Group) async {
    let userId = app.session?.user.id ?? UUID()
    homeCoordinator = HomeCoordinator(...)  // fetches async let upcoming

    profileCoordinator = ProfileCoordinator(...)
    await refreshMemberDirectory(for: group.id)

```

### Veredicto: **"Funciona, pero falta pulido"**
- ✓ State management es correcto (@Observable @MainActor)
- ✓ Persiste correctamente
- ✓ Fallback a first group si stale
- ✗ Sin loader visual durante rebuild
- ✗ Switcher no es discoverable desde todas las tabs
- ✗ Sin confirmación visual "you are on group X"

## 7. Coordinators

### Patrón estándar

```swift
@Observable @MainActor
final class FooCoordinator {
    private(set) var items: [Item] = []
    private(set) var isLoading: Bool = false
    private(set) var error: Error?
    
    func refresh() async {
        isLoading = true
        defer { isLoading = false }
        do {
            items = try await repo.list()
        } catch {
            self.error = error
        }
    }
}
```

### Coordinators auditados

| Coordinator | @Observable | @MainActor | async let | error | Nota |
|-----------|------------|-----------|-----------|-------|------|
| HomeCoordinator | ✓ | ✓ | ✓ (async let upcoming, myRSVP) | ✓ (EventError enum) | Bueno |
| InboxCoordinator | ✓ | ✓ | ✓ | ✗ | Falta error property |
| RulesCoordinator | ✓ | ✓ | ✓ | ✗ | Falta error property |
| MyFinesCoordinator | ✓ | ✓ | ✓ | ✗ | Falta error property |
| ProfileCoordinator | ✓ | ✓ | ✓ | ✗ | Falta error property |
| OpenVotesCoordinator | ✓ | ✓ | ✓ | ✗ | Falta error property |
| EventDetailCoordinator | ✓ | ✓ | ✓ (async let rsvps) | ✗ | Complejo, falta error |

### Problemas

**1. Inconsistent error handling**
- HomeCoordinator: `error: EventError?`
- Otros: No tienen error property
- Vistas no muestran .error state

**2. Refresh siempre es full reload**
- No hay partial updates
- No hay "incremental load" / pagination (excepto GroupHistoryCoordinator)
- Cada refresh causa flicker

**3. Task cancellation**
- No visto `Task` cancellation en coordinators
- ¿Qué pasa si la view desaparece durante async fetch?

**4. Cache TTL**
- HomeCoordinator: cacheTTL = 5 min
- Otros: no cachean

### Veredicto: **SÓLIDO PERO INCOMPLETO**

✓ Patrón @Observable @MainActor correcto
✓ async let para parallel fetches
✗ Error handling inconsistente
✗ Sin cache strategy (excepto HomeCoordinator)
✗ Sin refresh loading indicator cross-coordinator

## 9. Recomendaciones para Fase 0 Redesign

### 1. **Unify Loading States**
Reemplazar todos los bare `ProgressView()` con patrón consistente:
```
IF initial load (coordinator.items.isEmpty && isLoading):
    LoadingStateView(variant: .list | .card | .detail)
ELSE IF refresh (isRefreshing && !items.isEmpty):
    ScrollView { items } + refresh spinner header
ELSE IF error:
    ErrorStateView(message:, action: { retry() })
ELSE:
    content
```

**Escala:** 8 vistas principales (HomeView, Inbox, Rules, Profile, Events, Fines, Votes, Feed)
**Esfuerzo:** 2 sesiones (1 pattern, 1 rollout)

### 2. **Make Group Switcher Discoverable**
Opción A: Tab bar chip (smallest group name + tap → sheet)
Opción B: Sticky top nav bar (like Slack)
Opción C: Floating action button (pero ya hay crear evento FAB)

**Recomendación:** Opción B (sticky nav) — sigue Apple Sports aesthetic.

**Escala:** Tab bar refactor
**Esfuerzo:** 1 sesión

### 3. **Consolidate Component Library**
- Audit todas las vistas por custom components
- Extraer card/row variants a DesignSystem/Components
- Unificar spacing/colors via tokens (no hardcoded values)

**Escala:** Full codebase
